# Ceres v2: Action Head Training and Use in Search — preliminary results


## Context

The Ceres chess engine version 2 uses Monte Carlo Graph Search (MCGS), a graph-structured variant of the familiar PUCT-guided tree search (as also used by DAG versions of Leela Chess Zero). At each node in the search graph, two estimates are extracted from a deep neural network: a policy distribution over legal moves (which moves to explore) and a value estimate for the position (what the game is worth). The base Ceres network is not itself trained by self-play: it is derived from supervised learning against Lc0 training-game targets (policy distributions and W/D/L outcomes from completed self-play games) — a pragmatic choice that decouples network training from self-play data generation.

This note describes an addition to the Ceres v2 architecture: an action head — a network output that produces a per-move value distribution in parallel with the policy — together with the machinery that trains it post-hoc from an already-trained base network and uses it inside PUCT. The approach builds on some related ideas that have been percolating in the community (see Related Work below) and contributes an open and concrete implementation that appears to demonstrate some modest Elo gains and a direction for further enhancements.


## The gap in vanilla PUCT

The standard PUCT selection rule, in the form used by AlphaZero and most of its descendants, scores each child \*a\* of a node \*s\* by

U(s, a) = Q(s, a) + c\_puct · P(s, a) · √(Σ\_b N(s, b)) / (1 + N(s, a))

where P comes from the policy head, N is the visit count, and Q is the backed-up mean value from sub-tree visits. The challenge is what to do when N(s, a) = 0, i.e. when a child has not yet been expanded: Q(s, a) does not exist. The common approach is First-Play Urgency (FPU), a heuristic that fills in a placeholder:

Q\_FPU(s, a) = Q(s) − k · √Σ\_b P\_visited(s, b)        (the "reduction" variant)

or, more simplistically, a fixed constant. FPU works, but it is unsatisfying: every unvisited child receives the same scalar, regardless of what move it represents. The policy prior P(s, a) is the only per-move signal, but it is an estimate of the teacher's MCTS visit distribution — a move-selection policy rather than a per-move value.

The conceptually natural fix is to ask the network directly: for each legal move, what is your estimate of the value after playing it? That is a state-action value function Q(s, a) — a central object throughout reinforcement learning. The idea itself is not new (see \*Relation to prior work\*, below). Described here is a specific and simple post-hoc recipe for adding such a head to an already-converged supervised network, and its use as a drop-in replacement for FPU in MCGS.



## Training the action head



### Architecture

Ceres networks follow a two-stage architecture: a transformer encoder trunk operating on the 64 squares produces 
contextual per-square embeddings, and a collection of task-specific heads (policy, value, moves-left, 
uncertainty, …) project these embeddings to their respective outputs. Between the trunk and the specialised heads 
sits a small stack of shared head layers (a per-square linear "pre-map" and a flattening/mixing linear) that produces a compact feature vector consumed by all heads.



The action head we attach is a two-layer MLP taking this shared feature vector as input and projecting to a 
fixed-size output shaped \[92 × 3], i.e. 92 (maximum) move slots, each with a (W, D, L) triple. 



During retraining the encoder trunk is frozen. The shared head layers are not frozen; instead they receive 
a modest learning-rate tempering, so they can adapt slightly to the new auxiliary loss without drifting 
away from the regime in which the existing heads were trained. 
In practice the effect on the pre-existing heads falls within noise on policy and value metrics.



### Training targets



Targets for the action head are generated by a preprocessing step that amounts to a one-ply rollout from a 
fixed teacher. For each source position \*s\* — drawn from the same Lc0-derived training data 
used to train the base network — the training-data generator enumerates all legal moves \*a\*, 
plays each one to obtain the child position \*s′\*, and asks the frozen base network's value head for its 
(W, D, L) estimate of \*s′\*. That triple becomes the target for move \*a\*'s slot in the action head's output 
for position \*s\*.



The training loss is categorical cross-entropy between the head's per-move logits and the (W, D, L) 
target distribution, summed only over the legal moves. A typical run consumes roughly 50 million 
source positions and takes about one hour on four H100s; most of the compute goes to the inference pass
that generates the targets, not the gradient updates themselves.



### Why the targets should be noisier than the parent's value



The action head outputs are significantly less accurate than those of the base value head, 
with a correlation of approximately 0.85. This relatively low correlation is to be expected: 
the encoder stack was specifically pretrained with the value target, whereas the action head is 
trained with only the head parameters updated. Furthermore, the value head is only burdened with 
estimating a single W/D/L outcome, whereas the action head must model distributions over all 
legal moves. Even so, 0.85 is high compared to the 0.00 correlation offered by the usual FPU alternative.



## Integration into search

The action head is wired into PUCT under a UCI-visible flag:

&#x20;   setoption name EnableActionHead value true

When this flag is set, the MCGS FPU mode switches from the default reduction formula to an 
ActionHead mode (for both root and non-root nodes). The search modification itself is straightforward. 
For an unvisited child \*i\* of node \*s\*, in place of the reduction FPU,


Q\_FPU(s, i) = − ( A\_W(s, i) − A\_L(s, i) + ε )


where A\_W and A\_L are the action head's win and loss probabilities for move \*i\* 
(so the quantity in parentheses is the scalar expected value V = W − L); ε is a small optimism constant 
(0.10 in the current code), playing the role that reduction's exploration term usually does. 
The U (exploration) term of the PUCT formula is unaffected.


The rest of the PUCT arithmetic is unchanged. Once a child accumulates real visits, 
its Q is the mean of backed-up values as usual; the action head's role is strictly 
as a per-move prior on Q for the unvisited children, obviating FPU.



## Implementation cost

In the per-edge header of each MCGS node we added two FP16 fields — the per-move action value 
A\_V = W − L and a currently-unused slot for a per-move action uncertainty — for 4 additional bytes per edge. 
On real search trees this works out to around 150 bytes per node amortised. 
The head is evaluated once, as part of the regular NN call that produces policy and value, 
so there is no additional inference overhead, and wall-clock search overhead is negligible.



## Impact on Child Selection


FPU's failure mode is that it treats all unvisited children identically, 
letting PUCT's U-term (which weights by policy) do all the discriminating. 
The action head breaks that symmetry: now each unvisited child carries its own Q estimate, 
and PUCT's first exploratory visits are directed by a potentially richer signal. 
Direct comparison of the policy prior and the action-head values for the same children 
suggests the Q estimates carry genuinely additional information: they discriminate more 
finely between moves of similar or identical policy probability, and are less sensitive to 
artifacts of the teacher's data-generation process — for example, the coarse quantization of 
MCTS visit counts at low simulation budgets or particular choices of hyperparameters such as policy temperature.



## Empirical findings

Self-play testing has shown a consient modest Elo benefit — roughly +15 to +30 Elo depending on configuration 
— from enabling the action head. A representative data point: at a 300s + 5s time control on H100, 
Ceres-with-action-head versus Ceres-without on UHO-style openings yielded +27 ± 12 Elo. 
Preliminary tests suggest the Elo benefit is relatively constant across time controls 
(e.g. the same Elo difference at 30s + 0.5s). However quite possibly the Elo benefit would be more modest against
an opponent with a dissimilar search algorithm (such as alphabeta engines). Search speed is nearly unchanged.



## Availability

A first version of the feature is available on the Ceres GitHub under the tag `TCEC_S29_FRD`, 
and will be enabled for the Season 29 TCEC FRD event. 
The accompanying network, C3-768-30, is a fresh supervised-learning run: 30 encoder layers of dimension 768 
for a total of approximately 300 M parameters, trained on approximately 9.2 billion positions from 
Lc0 training games. Beyond the action-head addition, the main novelty in C3-768-30 is support for Chess960, 
made possible by the roughly 4% of the Lc0 training data drawn from Chess960 self-play games.



## Relation to prior work

Dedicated per-move value heads are not a new idea. 
KataGo's "three-head" work experimented with short-term and long-term Q heads alongside the main value head in Go, and state-action value functions have long been central to the broader RL literature. The Lc0 community has also discussed possible ways to generate action data that could be utilized as an FPU replacement. A recent attempt in the academic literature in the Go setting is QZero ("Mastering the Game of Go with Self-play Experience Replay", Liu \& Wang, 2026), which trains a single Q-value network via entropy-regularized Q-learning and forgoes search during training entirely. The work here is considerably more modest: Q-values are an auxiliary head grafted onto an otherwise conventional AlphaZero-style architecture, and are used only to improve PUCT's handling of unvisited children.


The approach is driven partly by practical considerations. 
Training an action head as part of the base network from scratch would require
per-move value targets across all legal moves, which neither self-play RL 
(which only backs up the move actually played) nor the existing Lc0 supervised data contain. 
Grafting the head on afterwards sidesteps that gap: it extracts an extra useful signal out of
the existing trained network.


It is hoped this work establishes that such an approach is readily feasible and 
appears to yield moderately significant Elo gains.



## Future work

Much remains to further investigate:

\- Broader testing of the Elo effect across time controls, opponents, and opening books.

\- Ablations that cleanly separate the action head's information contribution from any incidental effects.

\- Hyperparameter tuning of the PUCT integration

\- Alternative action-head architectures.

\- Extended and more diverse training (more positions, blended targets from other networks or engines).

\- An explicit action-head uncertainty head, which would let the selector weight the per-move prior by the head's own confidence.

\- Letting the encoder trunk itself participate in fine-tuning during action-head training.

\- Porting the same idea into Lc0 or other MCTS chess engines for independent validation.
